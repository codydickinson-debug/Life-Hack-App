Security Overview

How Ascend protects your data, end to end

✓ Local-first   ✓ Encrypted in transit & at rest   ✓ Per-device auth

This page is for anyone who wants to know exactly how Ascend handles your data — security researchers, privacy-minded users, and reviewers from partners like Plaid. It complements the Privacy Policy (which covers what data exists) by explaining how that data is protected.

Threat model in one sentence. Most of your data never leaves your device. The data that does leave (encrypted Plaid access tokens, AI chat snapshots) is gated by per-device authentication, encrypted with keys held only on the server, and transmitted exclusively over TLS to vendors with established security programs.

1. Data storage

1.1 On your device

All Ascend application data — habits, goals, plans, transactions, notes, settings — is written to your browser's localStorage. By default it's plaintext (browser-protected). You can opt into client-side AES-GCM encryption with a passphrase: the key is derived via PBKDF2-SHA256 with 600,000 iterations and held only in memory. Auto-lock re-prompts after a configurable idle period. Forgotten passphrase = unrecoverable; there is no backdoor.

1.2 On the backend (only if you connect a bank)

When you link a bank via Plaid, the Plaid access token is sent to a Cloudflare Worker that Ascend operates. Before being written to Cloudflare KV, the access token is AES-GCM encrypted with a server-side ENCRYPTION_KEY secret. KV records are keyed by your per-device user identifier; no other user can read your records.

A KV breach yields ciphertext that is useless without the server-side encryption key. The encryption key is a Cloudflare Worker secret — it is never written to source, KV, logs, or accessible from any HTTP path.

2. Encryption in transit

  • The web app is served over HTTPS only. Strict-Transport-Security: max-age=63072000; includeSubDomains; preload is set on every response.
  • All backend calls (Cloudflare Worker, Vercel Python API) require HTTPS; HTTP requests are rejected at the edge.
  • Plaid Link runs in an iframe served from plaid.com over TLS 1.2+. Bank credentials you enter never touch Ascend's code or backend — they're transmitted directly from Plaid Link to Plaid.

3. Authentication & access control

3.1 Device enrollment

Each device enrolls separately. On first use, the device calls POST /enroll with a shared enrollment key; the backend mints a per-device clientSecret, stores SHA-256(clientSecret) in KV keyed to the device's user ID, and returns the secret to the device. After enrollment, every subsequent request must carry the device's own Authorization: Bearer <clientSecret>. The shared enrollment key is no longer used.

A leaked per-device secret only impersonates one device. The user can revoke from the backend by deleting that device's KV record (Settings → "Revoke this device" — wipes the device's enrollment).

3.2 Backend access

  • The Cloudflare Worker is deployed exclusively by the operator named on the Privacy Policy. Cloudflare account access is protected by 2FA.
  • KV is reachable only from within the Worker. There is no admin dashboard exposed to the public internet.
  • Source code lives in a public GitHub repository; no production secrets exist anywhere in source. All sensitive values (PLAID_CLIENT_ID, PLAID_SECRET, ENCRYPTION_KEY, ANTHROPIC_KEY, ENROLLMENT_KEY) are Cloudflare Worker secrets, set via wrangler secret put.
  • The Vercel Python backend (which serves the static site and proxies Yahoo Finance + Anthropic) uses environment variables managed in the Vercel dashboard with role-based access.

4. Plaid-specific protections

  • Access token encryption at rest: AES-GCM with a server-held key. See §1.2.
  • Webhook signature verification: Every Plaid webhook is signed with ES256 JWT. The backend verifies the signature against Plaid's published key set, validates the body hash matches the signed claim, and enforces a 5-minute replay window. Unverified or replayed webhooks are rejected.
  • Per-invocation Plaid call budget: Each sync request has a cap on outbound Plaid calls to prevent runaway loops from burning Plaid quota.
  • Read-only: Ascend never invokes any Plaid endpoint that would move money. Bank sync is strictly Transactions / Auth / Identity / Liabilities / Investments read flows.
  • Disconnect is real: When you disconnect a bank, the encrypted access token is deleted from KV within minutes and the upstream Plaid item is revoked.

5. Logging & monitoring

  • The Worker emits structured per-request access logs (route, status code, latency, SHA-256(userId) truncated to 12 chars, hashed IP). User IDs are hashed in logs so log access doesn't leak account identifiers.
  • Per-user audit logs (last 200 events) are stored in KV and exposed back to the user via a /audit endpoint — so users can inspect what their own enrolled device has done.
  • No transaction details, dollar amounts, account numbers, or AI chat content are ever logged.
  • Cloudflare's built-in observability tracks request volume and error rates for the operator. Vercel's analytics tracks deployments only — no end-user behavioral analytics.

6. Rate limiting & abuse controls

  • /enroll — 5 enrollments per IP per hour, 25 per IP per day; global cap of 500/day across all IPs (defense in depth against credential-stuffing botnets).
  • Plaid endpoints — per-user rate limits to bound Plaid quota usage.
  • AI proxy — per-user daily message cap (default 50), per-minute burst cap (default 10), to bound Anthropic spend.
  • Account deletion — limited to 2/hour and 3/day per device to prevent accidental mass-wipe scripts.

8. Vulnerability disclosure

If you've found a security issue, please report it privately first.

  • Preferred: Open a GitHub Security Advisory. Stays private until we coordinate.
  • Fallback: Email codydickinson@autopalsusa.com with subject prefix [security].
  • Machine-readable contacts: /.well-known/security.txt (RFC 9116).
  • Policy detail: SECURITY.md covers scope, out-of-scope, response timelines, and known design choices.

Response targets: acknowledgement within 7 days, fix within 30 days for medium severity, within 7 days for high severity.

9. Incident response

If a security incident affects user data, the operator will:

  • Disable the affected code path or revoke the affected credentials within hours of confirmation.
  • Notify affected users within 72 hours via the in-app "What's new" sheet and (where applicable) email.
  • Publish a post-incident write-up in SECURITY.md with timeline, root cause, and remediation.
  • Rotate all relevant secrets and re-issue per-device credentials if any were exposed.

10. What we don't claim

We're not SOC 2 certified, ISO 27001 certified, or PCI-DSS compliant. Ascend is operated by an individual, not a company with a dedicated security team. The mitigations above are real — encryption, per-device auth, vendor selection, disclosure process — but they exist within the constraints of a small operation. The vendors that handle the highest-risk data (Plaid for bank credentials, Cloudflare for token storage, Anthropic for AI) carry their own SOC 2 Type II certifications, which transitively covers the riskiest parts of the stack.

If you need a counterparty with formal security certifications, Ascend may not be the right fit, and that's fair.
